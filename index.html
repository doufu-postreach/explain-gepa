2. The problem: RL is throwing away signal

To adapt a large language model to a downstream task — answering multi-hop questions, writing GPU kernels, following finicky formatting rules — the modern playbook is reinforcement learning. GRPO (Group Relative Policy Optimization) is a popular instance: roll out the model many times, score each rollout with a reward, and nudge the weights toward higher-reward behavior.

This works. It is also extraordinarily wasteful. The paper notes that "recent works typically use up to hundreds of thousands of rollouts for training." A rollout usually means a full LLM trajectory plus tool calls plus an evaluator pass. They cost real money and real time.

The deeper problem: a rollout is full of information — the chain of thought, the queries, the tool outputs, the evaluator's complaints. RL collapses all of that into one number. We are using a high-bandwidth signal to compute a 32-bit summary and then learning from the summary.

3. The core insight: rollouts are language, so reflect on them

Modern AI applications are compound systems: a question goes through a "rewrite query" module, then a "retrieve documents" tool, then a "summarize" module, then an "answer" module. Each module is driven by a prompt. When something fails, the failure is usually traceable to a specific module, and the trace is literally written in English.

GEPA's bet: instead of training weights, treat each module's prompt as a gene, run the system, read the trace, and have an LLM reflect on what went wrong and propose a smarter prompt. Then keep doing it.

Example trace from a HotpotQA second-hop module
[INPUT] question: "What is the population of the parish where the volcano Pico Ruivo is located?" [INPUT] summary 1: "Pico Ruivo is the highest peak on the island of Madeira, in the parish of Santana." [OUTPUT] query 2: "Santana parish population" [METRIC] score = 0.0 (retrieval missed; correct answer wanted total Madeira archipelago population) [FEEDBACK] "Retrieved doc didn't contain the answer. The question asked about the broader region (Madeira), but the query targeted only the small civil parish (Santana). Consider whether the question's scope matches your query's scope."

An RL system would see this trace and update weights based on reward = 0. GEPA passes the whole trace to a reflection LLM, which reads it and rewrites the prompt to add: "If the question is about a larger region, your query should target that wider region (e.g., 'Madeira archipelago population'), not the immediate sub-locale."

4. The algorithm — a genetic algorithm with three twists

Before opening the stepper, it helps to anchor the mental model. GEPA is a genetic algorithm. That phrase has a specific meaning that lines up cleanly with how GEPA works:

Genetic algorithm vocabularyWhat it means in GEPA
Individual / candidateA complete set of prompts, one per module of the compound system
Population / pool (P)The list of candidates GEPA has accumulated so far
FitnessScore(s) on the validation set D_pareto — but a vector, not a scalar
SelectionPick a parent from the pool to mutate. GEPA samples from the Pareto front.
MutationAn LLM reads the parent's failure traces and writes a new prompt for one module
Crossover"System-aware merge": combine two lineages by taking the best version of each module from each
AcceptanceRun the offspring on a fresh minibatch; keep it only if it improved over the parent

What's different from a textbook GA:

Reflective
Mutation isn't random bit-flips — it's an LLM reading what failed and rewriting prose
Pareto-based
Selection isn't "highest fitness" — it samples from candidates that uniquely dominate at least one task instance
Trace-grounded
Fitness comes with a feedback string, not just a number — that's what mutation reads

Show pseudocode (Algorithm 1, from the paper) 
function GEPA(Φ, D_train, μ, μ_f, B):
    # Split data
    D_feedback, D_pareto ← split(D_train)

    # Pool of candidates and their per-instance scores
    P ← [Φ]                                    # pool starts with the base system
    S ← [score(Φ, D_pareto, μ)]                # row of per-instance scores

    while budget B not exhausted:
        k     ← SELECTCANDIDATE(P, S)          # Pareto-weighted sampling
        j     ← SELECTMODULE(P[k])             # round-robin across modules
        M     ← sample minibatch from D_feedback

        # Run the candidate, gather rich feedback
        out, fb ← run(P[k], M, μ_f)

        # Reflective mutation: an LLM rewrites prompt j
        π'_j  ← UPDATEPROMPT(P[k][j], M, out, fb)
        Φ'    ← copy P[k] with module j replaced

        # Local acceptance test on the same minibatch
        if avg_score(Φ', M) > avg_score(P[k], M):
            P.append(Φ')
            S.append(score(Φ', D_pareto, μ))   # full D_pareto re-scoring

    return argmax_avg(P, S)
1Snapshot pool
2Pick parent
3Pick module
4Run minibatch
5Reflect & mutate
6Accept?
7Update pool

If step 2 looks suspicious — "why are we ignoring C0?" or "what does it mean for C2 to win an instance?" — that's the cue to read the next section, which lays out the Pareto fundamentals from scratch.

5. Reflective mutation, hands-on

Mutation in GEPA is not random. The reflection LLM is given the current instruction, several (input, output, feedback) triples from a minibatch, and a meta-prompt asking it to write a better instruction. Below we walk through a real example from the paper (Figure 2). But first we have to explain what task this module belongs to — otherwise the example won't parse.

Setup: the multi-hop QA pipeline

The example comes from HotpotQA, a benchmark where every question requires combining facts from two different Wikipedia articles. A single search rarely returns both. So the system breaks the work into a pipeline:

Reading left to right: the user's question goes in, an LLM module turns it into a first search query, a retriever pulls a document, another LLM summarizes that document into a paragraph called "summary 1", and then — the highlighted module — looks at the question + summary 1 together and decides what's still missing, writing a second search query to fill the gap. Final answer module combines everything.

So "summary 1" is just the summarized text of the first retrieved document. It tells the second-hop module what we already know after one search. The second-hop module's job is to figure out what we don't know yet and ask for it.

Concretely, suppose the question is "What is the population of the parish where the volcano Pico Ruivo is located?":

  • First search: "Pico Ruivo" → returns the Pico Ruivo Wikipedia page.
  • Summary 1: "Pico Ruivo is the highest peak on the island of Madeira, in the parish of Santana."
  • Second-hop module's job: write a query that finds the population the question is asking for.

A naive second-hop query is "Santana parish population". The seed prompt produces exactly that — and it scores 0.0, because Wikipedia's article on Santana parish doesn't list the population number the answer key expects (it expects the broader Madeira figure). That's the failure pattern reflective mutation will fix.

6. Pareto: from economics to GEPA

Where the word comes from

Vilfredo Pareto was an Italian economist around 1900. He's most famous for the "80/20 rule" (80% of land owned by 20% of people), but his deeper contribution was the idea of Pareto efficiency: an allocation of resources where you cannot make anyone better off without making someone else worse off. That's a definition of "no obvious waste."

Multi-objective optimization borrows this directly. Suppose you're choosing between several solutions and you care about multiple objectives at once. We say:

Domination, formally. Solution A dominates B if
    (1) A is at least as good as B on every objective, AND
    (2) A is strictly better than B on at least one objective.

The Pareto front is the set of solutions that no other solution dominates. They are the rationally defensible choices — every other solution is strictly worse than something on the front.

Now click D — A beats it on both axes, so D is dominated and falls off the front.

From 2D to N task instances

Here's the pivot to GEPA. In a typical run, the validation set D_pareto has dozens of task instances — say 50. Each instance is its own dimension. So a candidate prompt-set is a point in 50-dimensional objective space, where the value on dimension i is the score on task instance i.

Domination still has the same definition. A prompt-set A dominates B if A ≥ B on every instance AND A > B on at least one. The Pareto front is the set of prompt-sets that nothing else dominates. The math is identical — only the dimensionality changed.

The crucial consequence. In high dimensions, the Pareto front gets very wide. With 50 instances, almost any "specialist" candidate that is uniquely best at even one instance gets onto the front. This sounds like a bug, but it's the feature: GEPA wants specialists in the breeding pool. They carry idiosyncratic genes that the next reflection might combine into something that dominates everywhere.

How GEPA exploits the front

Two implementation details that make this concrete:

  1. Remove dominated candidates from the front. If A's instance-wins are a strict subset of B's, A is redundant — drop it.
  2. Sample stochastically from the survivors with probability proportional to how many instances each one wins. A candidate that uniquely solves 5 instances gets picked 5× more often than one that solves only 1.

The big picture. A textbook genetic algorithm rewards "best-on-average" parents and gradually loses diversity. GEPA's Pareto-based selection rewards any candidate that's the best at something — preserving the diverse genes the reflective mutator needs to keep climbing. That's the entire conceptual leap: replace "highest fitness" with "on the Pareto front."

9. Sample efficiency, broken down

10. Feedback engineering: the new prompt engineering

GEPA depends on a feedback function µf that returns more than a score. The richer the feedback, the better GEPA performs. The paper distinguishes three feedback regimes for a multi-hop QA system:

Scalar
"score = 0.4"
RL-style. Worst signal.
Trace
Per-module inputs and outputs revealed. Reflection has something to chew on.
Module-level
Per-hop annotation: "doc 1 hit, doc 2 miss". Best signal.
The new craft: build evaluators that say what they checked. Compiler errors. Unit test names. Constraint pass/fail lists. Retrieval hit/miss tables. PII categories that leaked. Each line of feedback is a free training signal. The authors explicitly call this out as underexplored.

11. Limitations & open questions

  • Prompts vs weights. "The boundary between prompt-based and weight-based learning is not well understood." In data-rich regimes, weight updates may dominate. The two could be combined.
  • LoRA-only baseline. Full-parameter GRPO might close the gap (but at much higher cost).
  • Instructions only. GEPA optimizes the natural-language instruction; few-shot exemplars are out of scope here.
  • Validation-heavy. Most rollouts go to scoring candidates on Dpareto. Smaller or adaptively subsampled validation could squeeze more out of the budget.
  • Merge timing. System-aware merge helps on GPT-4.1 mini but hurts on Qwen3-8B at fixed hyperparams. When to invoke crossover is open.
  • Feedback engineering. What feedback channel teaches fastest is largely unstudied.
